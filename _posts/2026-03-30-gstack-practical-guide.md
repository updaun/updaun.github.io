# gstack 실전 활용 가이드: AI 코딩 에이전트를 팀처럼 쓰는 구체적인 방법

요즘 AI 코딩 도구를 쓰다 보면 결국 같은 문제를 만납니다. “코드는 빨리 나오는데, 방향성 판단/리뷰/QA/배포는 여전히 사람이 병목”이라는 문제입니다. `gstack`은 이 병목을 줄이기 위해, 단일 프롬프트 도우미가 아니라 역할 기반 워크플로우(기획, 엔지니어링 리뷰, 코드 리뷰, QA, 배포)를 slash command로 묶어둔 오픈소스 스킬팩입니다. 즉, “AI에게 코드 한 덩어리 생성 요청”이 아니라 “작은 팀처럼 일 시키는 순서”를 만드는 도구라고 보는 게 정확합니다.

gstack의 핵심은 기능이 많은 것이 아니라, **작업 순서를 강제해 결과 품질을 끌어올리는 구조**에 있습니다. 예를 들어 `/office-hours`로 문제정의를 압축하고, `/plan-eng-review`로 기술 리스크를 먼저 드러낸 뒤, 개발 후 `/review`와 `/qa`를 통과한 변경만 `/ship`으로 PR 생성까지 이어집니다. 이 흐름을 한번 고정해 두면 “오늘은 뭘 먼저 해야 하지?” 같은 컨텍스트 스위칭 비용이 크게 줄어듭니다.

## gstack를 어디에 쓰면 효과가 큰가?

이 도구는 특히 1인 개발자, 작은 팀 리드, 빠른 실험이 많은 스타트업 환경에서 효과가 큽니다. 이유는 간단합니다. 기획-개발-검증-출시의 빈틈이 사람 수가 적을수록 더 커지기 때문입니다. gstack은 각 단계에 체크리스트 성격의 스킬을 붙여서 “빨리 만들었는데 나중에 품질 이슈로 되돌아가는 비용”을 줄여줍니다.

반대로, 이미 엄격한 사내 파이프라인(아키텍처 리뷰 위원회, QA 전담 조직, 보안 게이트, 릴리즈 매니저)이 고도로 자동화돼 있다면 체감 이득이 상대적으로 작을 수 있습니다. 즉, gstack은 없던 프로세스를 처음 만들거나, 느슨한 프로세스를 표준화할 때 가장 강력합니다.

## 설치 방법 (가장 먼저 따라할 부분)

아래는 Claude Code 기준 가장 빠른 설치 방법입니다.

```bash
git clone --single-branch --depth 1 https://github.com/garrytan/gstack.git ~/.claude/skills/gstack
cd ~/.claude/skills/gstack && ./setup
```

이 방식은 사용자 전역 설치입니다. 보통 개인 워크플로우를 빠르게 검증할 때 가장 편합니다. 팀 레포에 포함해 함께 쓰고 싶다면, 전역 설치 후 레포의 `.claude/` 경로에 복제해 고정 버전처럼 운영할 수 있습니다.

Codex/Gemini CLI/Cursor 계열에서 쓸 때는 호스트를 지정하는 형태를 권장합니다.

```bash
# repo-local 설치 예시
 git clone --single-branch --depth 1 https://github.com/garrytan/gstack.git .agents/skills/gstack
 cd .agents/skills/gstack && ./setup --host codex

# user-global 설치 예시
 git clone --single-branch --depth 1 https://github.com/garrytan/gstack.git ~/gstack
 cd ~/gstack && ./setup --host codex
```

설치 후 스킬이 안 보이면 대부분 링크가 꼬인 경우라서 아래 한 줄로 복구됩니다.

```bash
cd ~/.claude/skills/gstack && ./setup
```

## 처음 1시간: 이렇게 써야 감이 온다 (실전 시나리오)

처음부터 모든 명령을 외우려 하지 말고, 딱 4개만 순서대로 써보면 됩니다. 첫째, `/office-hours`로 만들고 싶은 기능이 아니라 해결하려는 문제를 먼저 정리합니다. 둘째, `/plan-eng-review`로 기술 설계의 리스크와 테스트 범위를 확정합니다. 셋째, 구현 후 `/review`로 결함 가능성을 줄입니다. 넷째, `/qa <staging-url>`로 실제 브라우저 검증을 통과시킵니다.

이 4단계만 매일 반복해도 “아이디어는 많은데 배포 가능한 결과물이 적은 상태”에서 빠르게 탈출할 수 있습니다. 중요한 점은 `/review`를 코드 완성 후 형식적으로 돌리는 것이 아니라, **수정 직후 바로 다시 실행해 회귀를 짧은 루프로 끊는 습관**입니다.

## 핵심 명령 8개만 이해하면 거의 다 쓴다

아래 8개를 우선순위로 익히면 됩니다.

1. `/office-hours`: 문제정의와 우선순위 정렬
2. `/plan-ceo-review`: 제품/비즈니스 관점의 범위 점검
3. `/plan-eng-review`: 기술 설계, 실패 시나리오, 테스트 전략 정리
4. `/review`: 코드 변경 리뷰 및 이슈 탐지
5. `/qa <url>`: 실제 브라우저 기반 동작 점검
6. `/ship`: 테스트/리뷰/PR 준비를 한 번에 진행
7. `/careful`: 파괴적 명령 전 경고 레이어 활성화
8. `/freeze`: 수정 가능한 디렉토리 범위 제한

여기서 `/careful`, `/freeze`는 생산성 명령이라기보다 사고 방지장치입니다. 특히 급하게 핫픽스할 때 실수 비용이 큰 팀이라면 기본값처럼 켜두는 것을 권장합니다.

## 실제 하루 루틴 예시 (바로 복붙해서 운영 가능)

오전에는 기획/설계 루프를 짧게 가져가고, 오후에는 구현/검증/출시로 닫는 루틴이 효율적입니다.

```text
09:30  /office-hours            -> 오늘 문제를 1~2개로 압축
10:00  /plan-eng-review         -> 구현 범위와 리스크 고정
11:00  구현 1차
13:30  /review                  -> 결함 수정
14:30  /qa https://staging...   -> 사용자 흐름 검증
16:00  /ship                    -> 테스트 + PR 생성
17:00  피드백 반영 후 재검증
```

핵심은 “완벽한 코드”보다 “짧은 검증 주기”입니다. gstack은 이 검증 주기를 slash command로 습관화하게 도와주는 도구입니다.

## 운영 팁: prefix, 업그레이드, 텔레메트리

명령 이름 충돌이 있으면 `./setup --prefix`로 `/gstack-qa`처럼 네임스페이스를 붙여 쓰는 게 안전합니다. 반대로 짧은 명령이 좋다면 `./setup --no-prefix`로 `/qa`, `/ship` 형태를 유지할 수 있습니다. 이 설정은 저장되어 이후에도 유지됩니다.

업그레이드는 `/gstack-upgrade`로 처리하는 것이 가장 간단합니다. 스킬이 오래된 상태에서 이상 동작이 나는 경우가 잦기 때문에, 트러블슈팅 전에 먼저 업데이트를 확인하는 습관이 좋습니다.

텔레메트리는 기본값이 OFF이며, opt-in 방식입니다. 공유 시에도 코드/파일경로/레포명/프롬프트 같은 사용자 콘텐츠는 보내지지 않는다고 문서에 명시되어 있습니다. 팀 정책상 외부 전송이 민감한 환경이라면 기본 OFF를 유지하고 내부 가이드에 명시해 두면 됩니다.

## gstack을 도입할 때 흔한 실패 패턴

첫 번째 실패는 “명령은 돌렸지만 기존 습관대로 아무거나 먼저 만드는 것”입니다. 이 경우 도구 효과가 거의 없습니다. 반드시 `문제정의 -> 설계검토 -> 구현 -> 리뷰 -> QA -> 출하` 순서를 유지해야 합니다.

두 번째 실패는 `/qa`를 생략하는 것입니다. 코드 리뷰를 통과해도 실제 사용자 동선에서 깨지는 포인트(인증, 리다이렉트, 모바일 뷰포트, 비동기 타이밍)가 남습니다. gstack의 강점이 브라우저 기반 검증까지 한 흐름에 넣는 데 있으므로, `/qa` 생략은 절반만 쓰는 것과 같습니다.

세 번째 실패는 “한 번에 너무 많은 스킬을 다 쓰려는 것”입니다. 처음 2주 정도는 `/office-hours`, `/plan-eng-review`, `/review`, `/qa`만 반복하세요. 루틴이 몸에 붙은 뒤 `/ship`, `/canary`, `/setup-deploy`를 추가해도 늦지 않습니다.

## 결론: gstack은 기능 모음이 아니라 작업 운영체제다

gstack을 잘 쓰는 방법은 새로운 명령을 많이 아는 것이 아니라, 매일 같은 리듬으로 반복하는 것입니다. 즉, “좋은 프롬프트”보다 “좋은 작업 순서”가 더 중요합니다. 작은 팀처럼 일하고 싶다면, 오늘은 설치만 하지 말고 `/office-hours -> /plan-eng-review -> /review -> /qa`까지 한 사이클을 실제로 끝내보세요. 그 한 번의 사이클이 gstack 도입 효과를 가장 빠르게 체감하게 해줍니다.
